# mini_web_service_ver3

Выполнена в рамках практической работы для университета

## Что реализовано

1. Конфигурация читается из 3 источников в явном порядке приоритета:
`appsettings.json` -> переменные окружения (`PR3_`) -> аргументы CLI.

2. Выполняется ранняя валидация конфигурации до запуска веб-приложения.
При ошибках сервис не стартует и завершает запуск с понятной причиной.

3. CORS ограничен доверенными источниками (`TrustedOrigins`).

4. Включён rate limiting с разными политиками для чтения и записи:
`read` для чтения, `write` для изменений (создание/удаление).

5. Добавлены защитные заголовки ответа:
`X-Content-Type-Options`, `X-Frame-Options`, `Cache-Control`, `Pragma`.

6. Поддержаны два режима работы:
`Учебный` и `Боевой`.
Режим выбирается только настройкой (`App:Mode`) и влияет на строгость ограничений и детализацию ошибок.

## API

1. `GET /api/auto-parts` - список автозапчастей.
2. `GET /api/auto-parts/by-id/{id}` - получить по id.
3. `POST /api/auto-parts` - добавить автозапчасть.
4. `DELETE /api/auto-parts/by-id/{id}` - удалить автозапчасть.
5. `GET /api/mode` - текущий режим.

Пример JSON для создания:

```json
{
  "name": "Тормозной диск",
  "price": 3490.50
}
```

## Запуск

Требования:

1. .NET SDK 8.0+
2. Доступные NuGet-источники

Базовый запуск:

```powershell
dotnet run --project src/Pr3.ConfigAndSecurity
```

### Режим 1: Учебный

```powershell
dotnet run --project src/Pr3.ConfigAndSecurity -- --mode Учебный --readPerMinute 60 --writePerMinute 20
```

Ожидаемое поведение:

1. Более подробные клиентские сообщения об ошибках.
2. Более мягкие ограничения по rate limit относительно базовой настройки.

### Режим 2: Боевой

```powershell
dotnet run --project src/Pr3.ConfigAndSecurity -- --mode Боевой --readPerMinute 60 --writePerMinute 20
```

Ожидаемое поведение:

1. Минимально многословные сообщения для клиента.
2. Строгие лимиты строго по заданным значениям.

## Настройка через 3 источника

### 1) Файл

`src/Pr3.ConfigAndSecurity/appsettings.json`

### 2) Переменные окружения

Примеры:

```powershell
$env:PR3_App__Mode = "Боевой"
$env:PR3_App__TrustedOrigins__0 = "http://localhost:5173"
$env:PR3_App__RateLimits__ReadPerMinute = "60"
$env:PR3_App__RateLimits__WritePerMinute = "20"
```

### 3) CLI аргументы

Поддерживаемые ключи:

1. `--mode`
2. `--origins`
3. `--readPerMinute`
4. `--writePerMinute`

Пример:

```powershell
dotnet run --project src/Pr3.ConfigAndSecurity -- --mode Боевой --readPerMinute 100 --writePerMinute 30
```

## Набор проверок

Автотесты находятся в `tests/Pr3.ConfigAndSecurity.Tests`.

Покрываются сценарии:

1. Приоритет источников конфигурации.
2. Валидация некорректной конфигурации.
3. Остановка запуска при некорректных настройках.
4. CORS для доверенного и недоверенного источника.
5. Срабатывание rate limit (`429`).
6. Наличие защитных заголовков.
7. Различия поведения `Учебный/Боевой`.
8. Интеграционный цикл создания и удаления автозапчасти.

Запуск:

```powershell
dotnet build Pr3.ConfigAndSecurity.Template.sln
dotnet test Pr3.ConfigAndSecurity.Template.sln
```

## Критичные настройки и почему

Критичными считаются:

1. `TrustedOrigins`.
Если список пуст или содержит ошибочный URL, можно либо случайно открыть API всем, либо заблокировать корректные клиентские запросы.

2. `RateLimits.ReadPerMinute` и `RateLimits.WritePerMinute`.
Слишком высокие лимиты снижают защиту от перегрузки, слишком низкие приводят к отказам для легитимных пользователей.

3. `Mode`.
Неправильный режим может раскрыть лишние детали ошибок в production или, наоборот, ухудшить обучаемость и диагностику в учебной среде.

## Как обеспечено раннее обнаружение ошибок конфигурации

В `Program.cs` конфигурация считывается и валидируется до `app.Build()` и до открытия HTTP-порта.
Если валидация не проходит, приложение:

1. Выводит конкретные причины ошибок конфигурации.
2. Прерывает запуск (`InvalidOperationException`).

Это снижает риски, потому что сервис не стартует в опасном или непредсказуемом состоянии:

1. Не запускается с пустыми/невалидными `TrustedOrigins`.
2. Не запускается с некорректными лимитами.
3. Не доходит до продакшн-трафика с ошибочной конфигурацией.

Итог: ошибки конфигурации ловятся на старте, а не во время инцидента под нагрузкой.
